# 📊 Procurement Flow - Типовий процес закупівлі

## Фази закупівлі

### 1️⃣ ФАЗА: Планування (Planning)

**Що відбувається:**
- Визначення потреби
- Розрахунок кошторису
- Вибір способу закупівлі
- Підготовка тендерної документації

**Статус:** `PLANNING`
**Етап:** `td` (tender documentation)

**Ключові завдання:**
- [ ] Визначити предмет закупівлі
- [ ] Розрахувати кошторис
- [ ] Вибрати категорію (обладнання, послуги, роботи)
- [ ] Підготувати ТД
- [ ] Визначити термін подачі пропозицій
- [ ] Підготувати форми документів

**Рішення системи:**
- Генерація унікального ID: `UA-YYYY-MM-DD-XXXXX-a`
- Валідація бюджету
- Перевірка типових помилок в ТД
- Рекомендації щодо ризиків

---

### 2️⃣ ФАЗА: Оголошення (Publication)

**Що відбувається:**
- Публікація тендеру на Prozorro.ua
- Відправлення спповіщень
- Контроль дат

**Статус:** `PUBLISHED`
**Етап:** `td` → `participants` (preparation)

**Ключові дати:**
- Дата публікації
- Дата початку аукціону
- Дата закінчення подачі пропозицій

**Рішення системи:**
- Перевірка всіх обов'язкових документів
- Контроль термінів
- Генерація повідомлень учасникам

---

### 3️⃣ ФАЗА: Прием пропозицій (Bidding)

**Що відбувається:**
- Учасники подають пропозиції
- Система відслідковує дедлайни
- Генеруються спповіщення

**Статус:** `BIDDING`
**Етап:** `participants`

**Завдання системи:**
- Контроль дати закінчення подачі
- Облік кількості пропозицій
- Моніторинг цін
- Генерація alert'їв за 24 години до кінця

**Рішення системи:**
- Автоматичні нагадування
- Контроль порушень
- Логування всіх операцій

---

### 4️⃣ ФАЗА: Аукціон (Auction)

**Що відбувається:**
- Проведення аукціону (якщо передбачено)
- Визначення найменшої ціни
- Реєстрація результатів

**Статус:** `BIDDING` / `EVALUATION`
**Етап:** `participants` → `24h`

---

### 5️⃣ ФАЗА: Кваліфікація (Qualification)

**ЦЕ НАЙВАЖЛИВІШИЙ ЕТАП ДЛЯ СИСТЕМИ**

**Що відбувається:**
- Перевірка документів учасників
- Визначення відповідності вимогам ТД
- Формування повідомлень про усунення невідповідностей (24 години)
- Прийняття рішень про відхилення

**Статус:** `EVALUATION`
**Етап:** `qualification` → `24h` → `rejections`

**Завдання Qualification Workspace:**
1. **Document Checklist** - Перевірка наявності документів
   - Контракт-аналог
   - Акти виконаних робіт
   - Ліцензії
   - КЕП
   - Статут
   - Довідки

2. **AI Analysis** - Аналіз документів
   - Перевірка строків дії
   - Перевірка відповідності ТД
   - Виявлення невідповідностей
   - Класифікація ризиків

3. **24-Hour Generator** - Формування повідомлень
   - Автоматичне формування тексту
   - Встановлення пунктів ТД
   - Розрахунок дедлайнів
   - Контроль 24-годинного терміну

4. **Decision Queue** - Черга рішень
   - Список учасників для рішення
   - Рекомендації системи
   - Кнопки для прийняття рішення

**Типові помилки учасників:**
- ❌ Прострочена довідка
- ❌ Відсутній документ
- ❌ Невірна дата
- ❌ Відсутній КЕП
- ❌ Невідповідність суми
- ❌ Невідповідність ТД

**Рішення системи:**
- Автоматична перевірка через чек-лист
- AI-асистент для виявлення проблем
- Генерація тексту повідомлення
- Відстеження 24-годинного терміну
- Рекомендація щодо відхилення

---

### 6️⃣ ФАЗА: Рішення (Decision)

**Що відбувається:**
- Прийняття рішення про відхилення
- Визначення переможця
- Публікація результатів

**Статус:** `EVALUATION`
**Етап:** `rejections` → `winner`

**Завдання:**
- [ ] Розглянути пояснення учасників
- [ ] Остаточно перевірити документи переможця
- [ ] Прийняти рішення про відхилення
- [ ] Визначити переможця
- [ ] Опублікувати рішення

**Рішення системи:**
- Automation Rejection Engine
- Knowledge Base з прецедентів
- Рекомендації на основі AI
- Контроль 24-годинної перевірки

---

### 7️⃣ ФАЗА: Договір (Contract)

**Що відбувається:**
- Укладення договору
- Підписання КЕП
- Публікація
- Початок виконання

**Статус:** `COMPLETED`
**Етап:** `contract`

**Завдання:**
- [ ] Підготувати проект договору
- [ ] Підписати КЕП
- [ ] Опублікувати договір
- [ ] Контролювати виконання
- [ ] Контролювати додаткові угоди (max +20%)

**Рішення системи:**
- Контроль дат підписання
- Контроль публікації
- Моніторинг додаткових угод
- Контроль не перевищення 20%

---

### 8️⃣ ФАЗА: Моніторинг (Monitoring)

**Що відбувається:**
- Контроль виконання договору
- Обробка моніторингів ДАСУ
- Обробка скарг

**Статус:** `COMPLETED`
**Етап:** `monitoring`

**Завдання:**
- [ ] Відслідковувати виконання
- [ ] Обробляти моніторинги
- [ ] Готувати відповіді
- [ ] Контролювати скарги

---

### 9️⃣ ФАЗА: Архівування (Archival)

**Що відбувається:**
- Збереження всіх документів
- Накопичення практики
- Оновлення Knowledge Base

**Статус:** `COMPLETED`
**Етап:** `archive`

**Завдання:**
- [ ] Архівувати документи
- [ ] Зберегти рішення
- [ ] Обновити Knowledge Base
- [ ] Генерувати звіт

---

## Ключові строки

| Етап | Строк | Важливо |
|------|--------|--------|
| Подача пропозицій | Зазвичай 14-30 днів | ⚠️ Критичне |
| Аукціон | 1 день | ⚠️ Критичне |
| Кваліфікація | 5 днів | ⚠️ Критичне |
| Повідомлення про усунення | 24 години | 🔴 КРИТИЧНЕ |
| Розгляд поправок | 24 години | 🔴 КРИТИЧНЕ |
| Укладення договору | 5 днів | ⚠️ Критичне |
| Публікація договору | Не пізніше дня укладення | ⚠️ Критичне |

## Контрольні списки

### Перед публікацією
- [ ] Перевірена ТД
- [ ] Визначені всі вимоги до учасників
- [ ] Підготовлені форми
- [ ] Встановлені дедлайни
- [ ] Перевірені ризики

### Під час оцінки пропозицій
- [ ] Перевірені всі документи
- [ ] Виявлені невідповідності
- [ ] Надіслані повідомлення за 24 години
- [ ] Відслідковані дедлайни відповідей

### Перед укладенням договору
- [ ] Остаточно перевірені документи переможця
- [ ] Готовий проект договору
- [ ] Підготовлена КЕП
- [ ] Контакти переможця верифіковані
